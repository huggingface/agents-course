# AI Agent のオブザーバビリティと評価

## 🔎 オブザーバビリティとは？

オブザーバビリティとは、ログ、メトリクス、トレースなどの外部シグナルを観察することで、AI Agent の内部で何が起きているかを理解することです。AI Agent においては、アクション、ツールの使用状況、モデルの呼び出し、レスポンスを追跡し、Agent のパフォーマンスをデバッグ・改善することを意味します。

![オブザーバビリティダッシュボード](https://huggingface.co/datasets/agents-course/course-images/resolve/main/en/bonus-unit2/langfuse-dashboard.png)

## 🔭 Agent のオブザーバビリティが重要な理由

オブザーバビリティがなければ、AI Agent は「ブラックボックス」です。オブザーバビリティツールは Agent を透明にし、以下のことを可能にします：

- コストと精度のトレードオフの把握
- レイテンシの測定
- 有害な言語やプロンプトインジェクションの検出
- ユーザーフィードバックの監視

言い換えれば、デモ段階の Agent を本番環境に対応させることができるのです！

## 🔨 オブザーバビリティツール

AI Agent 向けの一般的なオブザーバビリティツールには、[Langfuse](https://langfuse.com) や [Arize](https://www.arize.com) などのプラットフォームがあります。これらのツールは詳細なトレースを収集し、メトリクスをリアルタイムで監視するダッシュボードを提供することで、問題の検出やパフォーマンスの最適化を容易にします。

オブザーバビリティツールは、機能や特徴が大きく異なります。オープンソースのツールもあり、大規模なコミュニティがロードマップの策定や豊富なインテグレーションに貢献しています。また、オブザーバビリティ、評価、プロンプト管理など LLMOps の特定の側面に特化したツールもあれば、LLMOps ワークフロー全体をカバーするよう設計されたツールもあります。さまざまな選択肢のドキュメントを調べて、自分に合ったソリューションを見つけることをお勧めします。

[smolagents](https://huggingface.co/docs/smolagents/v1.12.0/en/index) などの多くの Agent フレームワークは、[OpenTelemetry](https://opentelemetry.io/docs/) 標準を使用してオブザーバビリティツールにメタデータを公開しています。これに加えて、オブザーバビリティツールは、急速に進化する LLM の世界でより柔軟に対応できるよう、カスタムのインスツルメンテーションを構築しています。使用しているツールのドキュメントを確認して、サポートされている内容を確認してください。

## 🔬 トレースとスパン

オブザーバビリティツールは通常、Agent の実行をトレースとスパンとして表現します。

- **トレース（Trace）** は、Agent タスクの開始から終了までの完全な流れを表します（ユーザーのクエリ処理など）。
- **スパン（Span）** は、トレース内の個々のステップです（言語モデルの呼び出しやデータの取得など）。

![Langfuse における smolagent トレースの例](https://huggingface.co/datasets/agents-course/course-images/resolve/main/en/bonus-unit2/trace-tree.png)

## 📊 監視すべき主要メトリクス

オブザーバビリティツールが監視する最も一般的なメトリクスを紹介します：

**レイテンシ：** Agent はどれくらい速く応答しますか？待ち時間が長いとユーザー体験に悪影響を与えます。Agent の実行をトレースすることで、タスク全体および個々のステップのレイテンシを測定する必要があります。例えば、すべてのモデル呼び出しに合計20秒かかる Agent は、より高速なモデルを使用したり、モデル呼び出しを並列実行したりすることで高速化できる可能性があります。

**コスト：** Agent の1回の実行にかかる費用はいくらですか？AI Agent はトークン単位で課金される LLM 呼び出しや外部 API に依存しています。ツールの頻繁な使用や複数のプロンプトは、コストを急速に増加させる可能性があります。例えば、Agent がわずかな品質向上のために LLM を5回呼び出す場合、そのコストが正当化されるか、呼び出し回数を減らしたり、より安価なモデルを使用できないか検討する必要があります。リアルタイムの監視は、予期しないスパイク（例：過度な API ループを引き起こすバグ）の特定にも役立ちます。

**リクエストエラー：** Agent が失敗したリクエストはいくつありますか？これには API エラーやツール呼び出しの失敗が含まれます。本番環境でこれらに対する堅牢性を高めるために、フォールバックやリトライを設定できます。例えば、LLM プロバイダー A がダウンした場合、バックアップとして LLM プロバイダー B に切り替えるといった対応です。

**ユーザーフィードバック：** ユーザーによる直接的な評価の実装は、貴重なインサイトを提供します。明示的な評価（👍高評価/👎低評価、⭐1〜5段階）やテキストコメントが含まれます。一貫してネガティブなフィードバックが寄せられる場合は、Agent が期待通りに動作していないサインであるため、アラートを設定すべきです。

**暗黙的なユーザーフィードバック：** 明示的な評価がなくても、ユーザーの行動は間接的なフィードバックを提供します。即座の質問の言い換え、繰り返しのクエリ、リトライボタンのクリックなどが含まれます。例えば、ユーザーが同じ質問を繰り返し投げかけている場合、Agent が期待通りに動作していないサインです。

**精度：** Agent はどの程度の頻度で正確または望ましい出力を生成しますか？精度の定義はさまざまです（例：問題解決の正確さ、情報検索の精度、ユーザー満足度）。最初のステップは、Agent にとっての成功の定義を明確にすることです。自動チェック、評価スコア、タスク完了ラベルを通じて精度を追跡できます。例えば、トレースを「成功」または「失敗」としてマーキングするといった方法です。

**自動評価メトリクス：** 自動評価を設定することもできます。例えば、LLM を使用して Agent の出力を「有用か」「正確か」などの観点でスコアリングできます。また、Agent のさまざまな側面をスコアリングするオープンソースライブラリもいくつかあります。例えば、RAG Agent 向けの [RAGAS](https://docs.ragas.io/) や、有害な言語やプロンプトインジェクションを検出する [LLM Guard](https://llm-guard.com/) などです。

実際には、これらのメトリクスを組み合わせることで、AI Agent の健全性を最もよく把握できます。この章の[サンプルノートブック](https://colab.research.google.com/#fileId=https://huggingface.co/agents-course/notebooks/blob/main/bonus-unit2/monitoring-and-evaluating-agents.ipynb)では、これらのメトリクスが実際のサンプルでどのように見えるかを紹介しますが、まずは一般的な評価ワークフローがどのようなものかを学びましょう。

## 👍 AI Agent の評価

オブザーバビリティはメトリクスを提供しますが、評価とは、そのデータを分析し（テストを実施して）、AI Agent がどの程度うまく機能しているか、どのように改善できるかを判断するプロセスです。言い換えれば、トレースやメトリクスを取得した後、それらをどのように使って Agent を評価し、意思決定を行うかということです。

定期的な評価が重要なのは、AI Agent は非決定的であることが多く、（アップデートやモデルの挙動の変化によって）進化しうるためです。評価がなければ、「賢い Agent」が実際にうまく機能しているのか、それとも性能が低下しているのかがわかりません。

AI Agent の評価には、**オンライン評価**と**オフライン評価**の2つのカテゴリがあります。両方とも価値があり、互いを補完し合います。通常はオフライン評価から始めます。これは Agent をデプロイする前に最低限必要なステップだからです。

### 🥷 オフライン評価

![Langfuse のデータセットアイテム](https://huggingface.co/datasets/agents-course/course-images/resolve/main/en/bonus-unit2/example-dataset.png)

オフライン評価は、管理された環境で Agent を評価することです。通常、ライブのユーザークエリではなく、テストデータセットを使用します。期待される出力や正しい動作がわかっているキュレーションされたデータセットを使用し、それに対して Agent を実行します。

例えば、数学の文章問題を解く Agent を構築した場合、既知の答えがある100問の[テストデータセット](https://huggingface.co/datasets/gsm8k)を用意するかもしれません。オフライン評価は通常、開発中に行われ（CI/CD パイプラインの一部にすることも可能）、改善の確認やリグレッションの防止に使用されます。利点は、**再現可能であり、正解データがあるため明確な精度メトリクスが得られる**ことです。ユーザークエリをシミュレーションして、Agent のレスポンスを理想的な回答と比較したり、上述の自動メトリクスを使用したりすることもできます。

オフライン評価の主な課題は、テストデータセットが包括的であり、関連性を維持し続けることです。Agent は固定されたテストセットでは良好なパフォーマンスを示すものの、本番環境ではまったく異なるクエリに遭遇する可能性があります。そのため、実際のシナリオを反映した新しいエッジケースや事例でテストセットを常に更新し続けるべきです。素早い確認用の小規模な「スモークテスト」ケースと、より広範なパフォーマンスメトリクス用の大規模な評価セットを組み合わせることが有効です。

### 🔄 オンライン評価

オンライン評価とは、ライブの実環境、つまり本番での実際の使用中に Agent を評価することです。オンライン評価は、実際のユーザーインタラクションにおける Agent のパフォーマンスを監視し、結果を継続的に分析することを含みます。

例えば、ライブトラフィックにおける成功率、ユーザー満足度スコア、その他のメトリクスを追跡することができます。オンライン評価の利点は、**ラボ環境では予測できないことをキャプチャできる**ことです。入力パターンの変化に伴う Agent の有効性の低下（モデルドリフト）を経時的に観察したり、テストデータに含まれていなかった予期しないクエリや状況を捉えたりすることができます。実際の環境で Agent がどのように振る舞うかの真の姿を提供してくれます。

オンライン評価では、前述のように暗黙的・明示的なユーザーフィードバックの収集や、シャドウテストや A/B テスト（Agent の新バージョンを旧バージョンと並行して実行し比較する）の実施が含まれることがあります。課題は、ライブインタラクションに対して信頼性のあるラベルやスコアを得ることが難しい場合があることです。ユーザーフィードバックや下流のメトリクス（例：ユーザーが結果をクリックしたかどうか）に依存する場合があります。

### 🤝 両方を組み合わせる

実際には、AI Agent の評価が成功するためには、**オンライン**と**オフライン**の手法を組み合わせることが重要です。定義されたタスクに対して Agent を定量的にスコアリングする定期的なオフラインベンチマークを実行しつつ、ベンチマークでは見逃してしまう問題を検出するためにライブ使用状況を継続的に監視する、といったアプローチが考えられます。例えば、オフラインテストではコード生成 Agent の既知の問題セットに対する成功率が向上しているかを確認でき、一方でオンライン監視ではユーザーが Agent が苦手とする新しいカテゴリの質問を始めたことを警告してくれます。両方を組み合わせることで、より堅牢な全体像が得られます。

実際、多くのチームは次のようなループを採用しています：_オフライン評価 → 新しい Agent バージョンのデプロイ → オンラインメトリクスの監視と新しい失敗事例の収集 → それらの事例をオフラインテストセットに追加 → イテレーション_。これにより、評価は継続的に改善されていきます。
