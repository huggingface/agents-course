# LangGraph の理解度テスト

`LangGraph` の理解度を簡単なクイズでテストしましょう！ここまでカバーしてきた重要な概念の復習に役立ちます。

これはオプションのクイズであり、採点はされません。

### Q1: LangGraph の主な目的は何ですか？
LangGraph が何のために設計されているかを最もよく表している文はどれですか？

- **A)** LLM を含むアプリケーションの制御フローを構築するためのフレームワーク ✅
  - 💡 LangGraph は、LLM を使用するアプリケーションの制御フローの構築と管理を支援するために特別に設計されています。
- **B)** さまざまな LLM モデルとやり取りするためのインターフェースを提供するライブラリ
  - 💡 これは、モデルとのやり取りに標準インターフェースを提供する LangChain の役割をよく表しています。LangGraph は制御フローに焦点を当てています。
- **C)** Tool Calling のための Agent ライブラリ
  - 💡 LangGraph は Agent と連携しますが、LangGraph の主な目的は「オーケストレーション」です。


---

### Q2: 「制御 vs 自由」のトレードオフにおいて、LangGraph はどこに位置しますか？
LangGraph の Agent 設計に対するアプローチを最もよく表している文はどれですか？

- **A)** LangGraph は自由度を最大化し、LLM がすべての意思決定を独立して行えるようにする
  - 💡 LangGraph は実際には自由よりも制御に重点を置いており、LLM ワークフローに構造を提供します。
- **B)** LangGraph は、意思決定に LLM の能力を活用しつつ、実行フローの強力な制御を提供する ✅
  - 💡 LangGraph は、Agent の実行を制御する必要がある場合に力を発揮し、構造化されたワークフローを通じて予測可能な動作を提供します。


---

### Q3: LangGraph における State の役割は何ですか？
LangGraph における State の最も正確な説明を選んでください。

- **A)** State は LLM からの最新の生成結果である
  - 💡 State は LangGraph においてユーザー定義のクラスであり、LLM が生成するものではありません。フィールドはユーザーが定義し、値は LLM が埋めることができます。
- **B)** State は実行中のエラーを追跡するためだけに使用される
  - 💡 State にはエラー追跡だけにとどまらない、はるかに広い目的があります。ただし、エラー追跡も有用です。
- **C)** State は Agent アプリケーションを流れる情報を表す ✅
  - 💡 State は LangGraph の中核であり、ステップ間の意思決定に必要なすべての情報を含んでいます。計算に必要なフィールドを定義し、ノードがその値を変更して分岐を決定できます。
- **D)** State は外部 API を使用する場合にのみ関連する
  - 💡 State は外部 API を使用するアプリケーションだけでなく、すべての LangGraph アプリケーションの基本です。


### Q4: LangGraph における Conditional Edge とは何ですか？
最も正確な説明を選んでください。

- **A)** 条件を評価して次に実行するノードを決定するエッジ ✅
  - 💡 Conditional Edge を使用すると、グラフが現在の State に基づいて動的なルーティングの決定を行い、ワークフローに分岐ロジックを作成できます。
- **B)** 特定の条件が発生した場合にのみたどられるエッジ
  - 💡 Conditional Edge は、入力ではなく出力に基づいてアプリケーションのフローを制御します。
- **C)** 処理を続行する前にユーザーの確認が必要なエッジ
  - 💡 Conditional Edge はプログラム的な条件に基づいており、ユーザーの操作を必要としません。


---

### Q5: LangGraph は LLM のハルシネーション問題にどのように対処しますか？
最適な回答を選んでください。

- **A)** LangGraph は LLM の応答を制限することでハルシネーションを完全に排除する
  - 💡 LLM からのハルシネーションを完全に排除できるフレームワークはなく、LangGraph も例外ではありません。
- **B)** LangGraph は LLM の出力を検証・確認できる構造化されたワークフローを提供する ✅
  - 💡 検証ステップ、確認ノード、エラー処理パスを含む構造化されたワークフローを作成することで、LangGraph はハルシネーションの影響を軽減するのに役立ちます。
- **C)** LangGraph はハルシネーションに影響を与えない
  - 💡 LangGraph のワークフローへの構造化されたアプローチは、速度を犠牲にしてハルシネーションの軽減に大きく貢献できます。
