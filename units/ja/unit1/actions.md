# アクション：エージェントが環境と対話するための手段

> [!TIP]
> このセクションでは、AIエージェントが環境と対話するために行う具体的なステップを探ります。
>
> アクションの表現方法（JSONやコードの使用）、Stop and Parseアプローチの重要性、そしてさまざまなタイプのエージェントについて紹介します。

アクションとは、**AIエージェントが環境と対話するために行う**具体的なステップです。

Webでの情報検索であれ、物理デバイスの制御であれ、各アクションはエージェントによって実行される意図的な操作です。

例えば、カスタマーサービスを支援するエージェントは、顧客データの取得、サポート記事の提供、または人間の担当者への問題の転送を行うことがあります。

## エージェントアクションの種類

アクションの実行方法が異なる複数のタイプのエージェントが存在します：

| エージェントの種類 | 説明 |
|------------------------|--------------------------------------------------------------------------------------------------|
| JSON Agent             | 実行するアクションがJSON形式で指定されます。 |
| Code Agent             | エージェントが外部で解釈されるコードブロックを記述します。 |
| Function-calling Agent | JSON Agentのサブカテゴリで、各アクションごとに新しいメッセージを生成するようにファインチューニングされています。 |

アクション自体はさまざまな目的を果たします：

| アクションの種類 | 説明 |
|--------------------------|------------------------------------------------------------------------------------------|
| 情報収集                  | Web検索の実行、データベースへのクエリ、ドキュメントの取得。 |
| ツールの使用              | API呼び出し、計算の実行、コードの実行。 |
| 環境との対話              | デジタルインターフェースの操作や物理デバイスの制御。 |
| コミュニケーション        | チャットを通じたユーザーとのやり取りや、他のエージェントとの連携。 |

LLMはテキストのみを扱い、テキストを使って実行したいアクションとツールに渡すパラメータを記述します。エージェントが正しく機能するためには、完全なアクションを定義するすべてのトークンを出力した後、LLMは新しいトークンの生成を**停止**する必要があります。これにより、LLMからエージェントに制御が戻され、JSON、コード、Function Callingのいずれの形式であっても、結果がパース可能であることが保証されます。


## Stop and Parseアプローチ

アクションを実装するための重要な手法の一つが、**Stop and Parseアプローチ**です。この手法により、エージェントの出力が構造化され、予測可能になります：

1. **構造化された形式での生成**：

エージェントは、意図するアクションを明確で事前に定義された形式（JSONまたはコード）で出力します。

2. **追加生成の停止**：

アクションを定義するテキストが出力されると、**LLMは追加のトークン生成を停止します**。これにより、余分な出力や誤った出力が防止されます。

3. **出力のパース**：

外部パーサーがフォーマットされたアクションを読み取り、呼び出すべきツールを判断し、必要なパラメータを抽出します。

例えば、天気を確認する必要があるエージェントは以下のように出力するかもしれません：


```json
Thought: I need to check the current weather for New York.
Action :
{
  "action": "get_weather",
  "action_input": {"location": "New York"}
}
```
フレームワークはこれにより、呼び出す関数名と適用する引数を簡単にパースできます。

この明確で機械可読な形式により、エラーが最小限に抑えられ、外部ツールがエージェントのコマンドを正確に処理できるようになります。

注：Function-calling Agentも同様に動作し、各アクションを構造化して、指定された関数が正しい引数で呼び出されるようにします。
これらのタイプのエージェントについては、今後のUnitでより詳しく掘り下げます。

## Code Agent

もう一つのアプローチは、*Code Agent*を使用することです。
その考え方は、**単純なJSONオブジェクトを出力する代わりに**、Code Agentが**実行可能なコードブロック（通常はPythonのような高水準言語）を生成する**というものです。

<img src="https://huggingface.co/datasets/agents-course/course-images/resolve/main/en/unit1/code-vs-json-actions.png" alt="Code Agents" />

このアプローチにはいくつかの利点があります：

- **表現力：** コードはループ、条件分岐、ネストされた関数などの複雑なロジックを自然に表現でき、JSONよりも高い柔軟性を提供します。
- **モジュール性と再利用性：** 生成されたコードには、異なるアクションやタスク間で再利用可能な関数やモジュールを含めることができます。
- **デバッグの容易さ：** 明確に定義されたプログラミング構文により、コードのエラーは検出と修正がしやすくなります。
- **直接的な統合：** Code Agentは外部ライブラリやAPIと直接統合でき、データ処理やリアルタイムの意思決定などのより複雑な操作が可能になります。

LLMが生成したコードの実行には、プロンプトインジェクションから有害なコードの実行まで、セキュリティリスクが伴う可能性があることに留意する必要があります。
そのため、デフォルトのセーフガードを統合した`smolagents`のようなAIエージェントフレームワークの使用が推奨されます。
リスクとその軽減方法について詳しく知りたい場合は、[こちらの専用セクションをご覧ください](https://huggingface.co/docs/smolagents/tutorials/secure_code_execution)。

例えば、天気情報の取得を任されたCode Agentは、以下のようなPythonスニペットを生成するかもしれません：

```python
# Code Agentの例：天気情報の取得
def get_weather(city):
    import requests
    api_url = f"https://api.weather.com/v1/location/{city}?apiKey=YOUR_API_KEY"
    response = requests.get(api_url)
    if response.status_code == 200:
        data = response.json()
        return data.get("weather", "No weather information available")
    else:
        return "Error: Unable to fetch weather data."

# 関数を実行して最終回答を準備
result = get_weather("New York")
final_answer = f"The current weather in New York is: {result}"
print(final_answer)
```

この例では、Code Agentは：

- **API呼び出しを通じて**天気データを取得し、
- レスポンスを処理し、
- print()関数を使用して最終回答を出力します。

この手法も**Stop and Parseアプローチに従っており**、コードブロックを明確に区切り、実行完了のシグナルを送ります（ここでは、final_answerの出力によって）。

---

アクションは、JSON、コード、またはFunction Callを通じて明確で構造化されたタスクを実行することで、エージェントの内部推論と現実世界での対話を橋渡しすることを学びました。

この意図的な実行により、Stop and Parseアプローチを通じて、各アクションが正確で外部処理に対応できることが保証されます。次のセクションでは、エージェントが環境からのフィードバックをどのように捕捉し統合するかを見るために、オブザベーション（Observation）について探ります。

その後、いよいよ**最初のエージェントを構築する準備が整います！**
